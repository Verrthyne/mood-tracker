# AGENTS.md

この文書は、本プロジェクトをCodexで継続開発する際の恒久的な作業ルールを定める。
現在の実装状況や既知の問題など、変化する情報は `PROJECT_STATUS.md` に記載する。

## 基本原則

- 既存機能と既存データを壊さない。
- 変更は依頼を満たすために必要な最小範囲に限定する。
- 無関係なリファクタリング、整形、命名変更を同時に行わない。
- 旧チャットや説明文より、現在のコードと実際の動作を優先する。
- 未確認事項を「確認済み」「正常」「実装済み」と記載しない。
- 事実、実行確認結果、未確認事項、推測を区別して報告する。

## 作業開始前

- `PROJECT_STATUS.md` と対象コードを読む。
- ファイル構成、Git状態、既存の未コミット変更を確認する。
- ユーザーの既存変更を無断で上書き、破棄、巻き戻ししない。
- 実装前に、少なくとも以下への影響を確認する。
  - 記録の新規作成、編集、削除
  - `localStorage` の保存キーと保存形式
  - 古い保存データの読み込み
  - CSV・JSONバックアップ
  - 各分析結果
  - クイック入力と詳細入力
  - スマートフォン表示
  - キーボード操作とアクセシビリティ
  - GitHub Pagesでの公開

## データ保護と互換性

- 保存キーを明示的な許可なく変更しない。
- 保存データの既存フィールドを無断で削除、改名、意味変更しない。
- 既存記録、追加指標、タグ、CBTメモを失う変更を行わない。
- 実データの削除、初期化、置換、移行を明示的な許可なく行わない。
- 古い保存データとの互換性を維持する。
- 保存形式を変更する場合は、事前に以下を提示して許可を得る。
  - 変更理由
  - 新旧形式
  - 移行処理
  - 失敗時の扱い
  - 復旧方法
- 移行処理は可能な限り追加的かつ再実行可能にし、旧データを自動削除しない。
- JSONインポートなど既存データを置き換える処理は、バックアップ、検証、確認手順を重視する。
- 公開版や旧URLに保存された実データを、テスト目的で使用しない。
- 動作確認用データは、実データと分離されたlocalhostなどの保存領域に作成する。

## 実装方針

- 現在の構成と設計を尊重し、必要性がない限り技術構成を変更しない。
- パッケージや外部ライブラリを、明示的な許可なく追加・更新しない。
- CDN、保存形式、公開方式などの外部依存を変更する場合は、影響を先に説明する。
- ユーザー入力やインポートデータをHTMLへ表示する場合は、適切にエスケープする。
- 削除や全置換などの破壊的操作は、誤操作と復旧可能性を考慮する。
- UIの見た目だけでなく、保存結果と再読み込み後の状態まで確認する。
- 分析機能を変更する場合は、名称、集計対象、計算式、境界値を明確にする。
- アクセシビリティの既存実装を後退させない。
- スマートフォン向けレイアウトと固定保存バーを壊さない。

## 実装後の確認

- 変更差分を確認し、依頼範囲外の変更がないことを確認する。
- JavaScriptの構文確認を行う。
- 変更内容に応じて、以下を回帰確認する。
  - 初期表示
  - クイック入力ON/OFF
  - スライダーと数値入力の同期
  - 新規記録
  - 既存記録の編集
  - 削除
  - 同日複数記録
  - 複数タグ
  - CBTメモ
  - 追加指標
  - 曜日、時間帯、タグ、外出比較、簡易相関
  - 時系列グラフ
  - CSV・JSONバックアップ
  - スマートフォン幅
  - キーボード操作とスクリーンリーダー向け状態通知
- 実行できなかった確認は、その理由とともに未確認として報告する。
- テストのために作成した一時サーバーやプロセスは終了する。
- 実装状況や既知の問題が変わった場合は、`PROJECT_STATUS.md` の更新案を提示する。

## Git・GitHub・公開

- 明示的な許可なしに以下を行わない。
  - commit
  - amend
  - rebase
  - 履歴書き換え
  - push
  - force-push
  - Pull Request作成
  - デプロイ
  - GitHub Pages設定変更
  - ブランチ作成・削除
  - tag・release作成
- Git履歴や設定に個人情報が含まれていないか注意する。
- メールアドレスなどの個人情報を、報告やログに必要以上に再掲しない。
- 公開版とローカル版の差分を、確認せずに同一とみなさない。
- GitHub Pagesへ反映する前に、公開対象ファイルと差分を明示する。

## 報告

作業完了時は、少なくとも以下を簡潔に報告する。

- 変更したファイル
- 変更内容
- 保存形式・既存データへの影響
- 実行した確認
- 確認できなかった事項
- 既知のリスク
- commit、push、デプロイの実施有無
